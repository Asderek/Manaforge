What this document is trying to do
This version explains the project in language that should feel familiar to someone used to classic 
PHP sites uploaded by FTP.
Whenever a modern term appears, the document translates it into something practical: what it 
is, why it exists, and what part of the old mental model it replaces.
It also adds a concrete email plan for account activation, password recovery, approval notices, 
and other messages the system must send.
Short project sentence
Mana Forge is a web application for building playtest-only Magic: The Gathering proxy sheets, 
saving projects online, sharing read-only deck views by link, and exporting printable PDFs.
Architecture in one line
GitHub Pages serves the site files, a Cloudflare Worker behaves like the backend script endpoint, 
Cloudflare D1 stores the data, Google reCAPTCHA reduces bot registrations, and Resend sends 
account and recovery emails.

2. What we are building in plain language
Mana Forge is not meant to be an “official-looking card factory”. It is a playtest workshop tool. The 
user imports or types a deck, chooses a simple proxy style, previews printable sheets, and exports a 
PDF meant for private testing at home.
The application also has accounts, because saved projects, moderation, and controlled sharing 
require us to know who owns what.
The user-facing features are:
Create and save proxy projects.
Paste or edit decklists.
Preview simplified proxy card layouts.
Generate printable PDF sheets.
Open a deck viewer through a shareable link.
Request an account, recover a password, and receive system emails when needed.
3. The architecture explained without buzzwords
Think of the system as two separate things working together:
1.The published site files: these are the pages, scripts, and styles that the browser downloads from 
GitHub Pages.
2.The trusted service: this is the Cloudflare Worker, which behaves like the backend and talks to 
the database and email provider.
The easiest mental picture
GitHub Pages is the “public_html” folder on the internet.
The Worker is the server-side script layer that you do NOT upload by FTP into the same folder.
D1 is the SQL database behind that script layer.
Resend is the external mail service used when the system must send real emails that arrive 
reliably.

4. Why Next.js is still acceptable even though the site is static
A static site does not mean “simple HTML only”. It means the published host is serving already-built 
files, not executing request-time page code. Next.js is still useful here because it organizes the 
interface, routing, and component structure during development, then exports the finished files for 
GitHub Pages.
Relatable version: we are using a modern site-building framework to prepare the files, but the final 
host behaves more like old-school file hosting than like a PHP application server.
Implications of this decision:
No PHP-like page code runs on GitHub Pages.
No database connection exists inside the published page files.
Every trusted action must go through the Worker.
Share pages must be designed so they still work as regular static routes plus query-string data.

6. Registration and account control - revised to be more practical
Because registration is moderated, the cleanest design is to avoid collecting a permanent password 
before approval. That keeps the “pending request” stage simpler and avoids storing credentials for 
accounts that may never be activated.
Recommended account flow:
3.A visitor opens the registration request form.
4.The visitor enters display name, email address, optional note, and completes Google reCAPTCHA.
5.The Worker verifies the reCAPTCHA token with Google.
6.If valid, the Worker stores a pending registration request in D1.
7.The system sends a “request received” email.
8.An administrator approves or rejects the request from the admin area.
9.If approved, the system sends an activation email with a one-time link.
10.The user opens the activation link, sets a password, and confirms the email address in the same 
flow.
11.After activation, the user can log in and use the application.
Why this is better than asking for a password in the first form
Mana Forge planning document - page 6We do not keep dormant passwords for people who may be rejected or may never return.
The approval email naturally becomes the first verified contact point.
The activation page can combine “verify email” and “create password”, which is easier for users 
and easier to explain.
7. reCAPTCHA in practical terms
reCAPTCHA is only for the public request form and any other public form that is likely to attract 
bots. It is not a login system and it is not a user identity system.
The old-school way to think about it: it is the modern version of adding a “prove you are not an 
automated script” gate before your form actually reaches the server logic.
Recommended version
Use Google reCAPTCHA v2 Checkbox as the default for registration requests. It is easier to explain, 
easier to test, and more visible to administrators than a score-only invisible system.
Implementation notes
The frontend loads Google’s reCAPTCHA widget and obtains a token when the user completes the 
challenge.
The browser sends that token to the Worker together with the registration form.
The Worker sends the token to Google’s verification endpoint.
Only if Google says the token is valid do we store the request.
The reCAPTCHA secret key stays only in Worker secrets, never in the browser code.

9. Email delivery - how to send valid account emails
Because GitHub Pages is only serving static files, email must be sent by the Worker. The most 
straightforward choice here is Resend, because there is an official Cloudflare Workers tutorial for it 
and a free plan suitable for low-volume transactional mail.
Recommended email setup
Email provider: Resend.
Sending domain: a real domain or subdomain such as mail.manaforge.example.
Main sender addresses: no-reply@..., support@..., and optionally admin@....
Backend integration point: the Worker calls Resend with an API key stored as a Cloudflare 
secret.
Tracking table: store an email log in D1 with message type, recipient, related user, send status, 
and provider response ID.
What “send valid emails” means in practice
Use a verified sending domain, not a random unverified sender address.
Publish the DNS records required by the provider: at minimum SPF and DKIM, with DMARC strongly  
recommended.
Send from consistent addresses such as no-reply@manaforge.example and 
support@manaforge.example.
Send plain-text and HTML content so the message is readable in more clients.
Keep links short, clear, and tied to one-time tokens or signed records.
Never send passwords by email.

Recommended token design
Generate a long random token on the backend.
Store only a hash of that token in D1, not the raw token.
Store the token type, related user, expiration time, and used_at timestamp.
When the browser returns with the token from the URL, the Worker hashes it again and checks the 
stored record.
Once used, mark it used so it cannot be replayed.
Relatable version: treat these tokens like temporary password-reset rows in a database table, not 
like values the browser can invent or trust on its own.

13. PDF generation and print safety rules
The PDF export remains part of the browser-side application. This keeps the backend simpler and 
avoids building a server-side rendering pipeline for printable sheets.
Mandatory content rules:
Every generated card face must visibly state PROXY or PLAYTEST.
The back face must be generic and must not mimic the official MTG card back.
Do not imitate holo stamps, rarity foil treatment, or any authenticity cues.
The export screen and the share viewer must show playtest-only notices such as “Not for sale” and 
“Not tournament legal”.

15. Decisions that keep the project easy to reason about
Keep GitHub Pages responsible only for serving files, never for trusted business logic.
Keep the Worker as the single place that knows secrets, permissions, token validity, and database 
writes.
Delay anything not required for the first useful version, such as social login, full card API 
enrichment, or public user profiles.
Prefer simple, visible flows over “magic” flows. An explicit activation email is easier to explain than 
hidden automatic account states.
Treat all share links as read-only views.

18. Final recommendation
Proceed with the stack exactly as planned, but explain it internally using the simpler model: “static 
files on GitHub Pages, trusted backend on Cloudflare Worker, SQL in D1, email through Resend, bot 
check through reCAPTCHA.” That description is technically honest and much easier for a PHP-era 
mental model to follow.
